## [0x00] Why this is one of the most useful stryke packages

DuckDB is an in-process analytical SQL engine. With this package, a
stryke one-liner gets:

```stryke
use DuckDB
p DuckDB::query "SELECT COUNT(*) FROM 's3://my-bucket/events/*.parquet'"
```

No import step, no schema declaration. DuckDB infers everything, vectorizes
the scan, and returns a row count. The same engine that powers Motherduck,
the same SQL surface as PostgreSQL, with the embedded-runtime ergonomics
of SQLite.

Pairs cleanly with [stryke-arrow](../stryke-arrow) (data pipeline) and
[stryke-parquet](../stryke-parquet) (file diagnostics). DuckDB closes the
"now what do I do with this data" loop ‚ÄĒ runs SQL on the file you just
inspected.

## [0x02] Quick start

```stryke
use DuckDB

# In-memory queries (default ‚ÄĒ every invocation starts fresh).
p DuckDB::query "SELECT 1 + 1 AS two, 'hi' AS s"

# Query a parquet file directly ‚ÄĒ no table to create.
my @rows = DuckDB::query "SELECT id, name FROM 'events.parquet' LIMIT 10"
@rows |> ep

# Aggregate a CSV.
my $count = DuckDB::query_scalar
    "SELECT COUNT(*) FROM read_csv_auto('orders.csv') WHERE total > 100"

# Hit remote files (requires httpfs extension; load it on connect).
my @rows = DuckDB::query
    "SELECT * FROM 'https://example.com/data.parquet' LIMIT 5",
    extensions => ["httpfs"]

# Persistent file db.
DuckDB::execute "CREATE TABLE users (id INT, name VARCHAR, score DOUBLE)",
                db => "app.duckdb"
DuckDB::execute "INSERT INTO users VALUES (?, ?, ?)",
                bind => [42, "alice", 1.5],
                db => "app.duckdb"

# Bulk load a parquet into a table.
my $r = DuckDB::import "events.parquet", "events",
                       db => "app.duckdb",
                       replace => 1
p "loaded $r->{num_rows} rows"

# Dump table back out.
DuckDB::export "events", "events.zstd.parquet",
               db => "app.duckdb",
               kind => "parquet",
               compression => "zstd"

# Stream large results without buffering on the stryke side.
DuckDB::query_stream "SELECT * FROM events",
    db => "app.duckdb",
    callback => sub ($row) { process $row }
```

## [0x03] Connection options

Every `DuckDB::*` op accepts `%opts` as its final argument. Connection
fields the cdylib understands (matching the v1 helper-binary flags):

```
db          ‚Üí path to a `.duckdb` file. Omit for `:memory:` (default).
session     ‚Üí name for distinct `:memory:` instances. Defaults to "_default";
              same-session calls share the same in-memory db.
read_only   ‚Üí 1 to open the file db RO
pragmas     ‚Üí \@stmts ‚ÄĒ `SET name=value;` strings to run on connect
extensions  ‚Üí \@names ‚ÄĒ `INSTALL <name>; LOAD <name>;` for each on connect
```

Inline:

```stryke
DuckDB::query "SELECT COUNT(*) FROM 'events.parquet'",
    extensions => ["httpfs"]

DuckDB::execute "INSERT INTO users VALUES (?, ?, ?)",
    bind => [42, "alice", 1.5],
    db   => "app.duckdb"
```

The cdylib caches one `duckdb::Connection` per `(db, session, read_only)`
tuple ‚ÄĒ `:memory:` databases persist across calls (the v1 helper binary
got a fresh empty `:memory:` every fork). Two calls with the same
`db => "app.duckdb"` share the same connection object.

Common extensions:

```
httpfs    HTTP / HTTPS / S3 file reads
aws       S3 with AWS-SDK auth
iceberg   Iceberg table format
delta     Delta Lake
spatial   geospatial functions
excel     .xlsx reader
```

## [0x04] API reference

### Read paths

```stryke
DuckDB::query         $sql, %opts ‚Üí @rows | hashref | meta-hashref
DuckDB::query_stream  $sql, %opts ‚Üí $count             # callback per row
DuckDB::query_one     $sql, %opts ‚Üí \%row | undef
DuckDB::query_col     $sql, %opts ‚Üí @values
DuckDB::query_scalar  $sql, %opts ‚Üí $value | undef
DuckDB::dump          $source, %opts ‚Üí @rows           # source = table | path | URL
```

`%opts`: `db`, `pragmas` (arrayref), `extensions` (arrayref), `read_only`,
`bind` (arrayref for `?` placeholders), `columnar`, `with_meta`, `limit`,
`callback` (stream only).

### DDL / DML

```stryke
DuckDB::execute     $sql, %opts ‚Üí { affected }
DuckDB::exec_file   $path, %opts ‚Üí { ok: true }
DuckDB::explain     $sql, %opts ‚Üí $plan_text          # opts: analyze => 1 for EXPLAIN ANALYZE
DuckDB::insert_many $table, $rows_aref, %opts ‚Üí $inserted_count   # single multi-row INSERT
DuckDB::appender    $table, $rows_aref, %opts ‚Üí $appended_count   # native Appender ‚ÄĒ fastest bulk load
DuckDB::appender_columns $table, $cols_aref, $rows_aref, %opts ‚Üí $appended_count   # native Appender into a column subset (rest take DEFAULT/NULL)
DuckDB::import      $path, $table, %opts ‚Üí { table, kind, source, num_rows }
DuckDB::export      $table, $path, %opts ‚Üí { table, kind, path, file_size }
DuckDB::copy_from   $table, $file, %opts ‚Üí { table, path, copied }   # COPY ‚Ä¶ FROM into an existing table; opts: kind => csv|parquet|json|auto
DuckDB::create_index $name, $table, $cols_aref, %opts ‚Üí { index, table, columns }   # opts: unique => 1, if_not_exists => 1
DuckDB::drop        $name, %opts ‚Üí { dropped, kind }   # opts: kind => table|view|index (default table), if_exists (default 1), cascade
DuckDB::drop_index  $name, %opts ‚Üí { dropped, kind }   # convenience for drop kind => index
DuckDB::attach      $file, $alias, %opts ‚Üí { alias, attach_path, read_only }   # ATTACH a db file under $alias; opts: attach_read_only, if_not_exists
DuckDB::detach      $alias, %opts ‚Üí { alias, detached }   # DETACH; opts: if_exists
DuckDB::update      $table, $set_href, $where?, %opts ‚Üí $affected   # UPDATE ‚Ä¶ SET ‚Ä¶ [WHERE]
DuckDB::delete      $table, $where?, %opts ‚Üí $affected               # DELETE FROM ‚Ä¶ [WHERE]
DuckDB::truncate    $table, %opts ‚Üí 1                 # DELETE FROM (empties the table)
DuckDB::upsert      $table, $row_href, %opts ‚Üí $affected | @rows   # INSERT ‚Ä¶ ON CONFLICT DO UPDATE
DuckDB::quote_ident $name ‚Üí $quoted               # ANSI double-quote: my col ‚Üí "my col"
DuckDB::unquote_ident $quoted ‚Üí $name             # inverse of quote_ident: "we""ird" ‚Üí we"ird (un-double, strip)
DuckDB::quote_literal $v ‚Üí $literal               # value literal: undef‚ÜíNULL, number‚Üíbare, string‚Üí'...' ('' doubled)
DuckDB::unquote_literal $quoted ‚Üí $value          # inverse (string form): 'O''Brien' ‚Üí O'Brien (un-double, strip)
DuckDB::quote_like $s ‚Üí $pattern_body             # escape % _ \ so $s matches literally in LIKE ‚Ä¶ ESCAPE '\'
DuckDB::unquote_like $pattern ‚Üí $literal          # inverse: recover the literal (100\%off ‚Üí 100%off); rejects an unescaped wildcard or dangling backslash
DuckDB::quote_qualified_ident $name ‚Üí $quoted     # main.my table ‚Üí "main"."my table"
DuckDB::parse_qualified_ident $name ‚Üí \@parts     # "main"."my table" ‚Üí ["main","my table"]; inverse of quote_qualified_ident
DuckDB::format_list \@elements ‚Üí $literal         # ["a","b"] ‚Üí ['a', 'b'] (DuckDB LIST literal)
DuckDB::parse_list $literal ‚Üí \@elements          # ['a', 'b'] ‚Üí ["a","b"]; inverse of format_list (numbers bare, '' un-doubles, comma-in-quotes literal)
DuckDB::format_in_list \@values ‚Üí $operand        # [1,"a",undef] ‚Üí (1, 'a', NULL) for `col IN (...)`; empty ‚Üí (NULL)
DuckDB::parse_in_list $operand ‚Üí \@values         # (1, 'a', NULL) ‚Üí [1,"a",undef]; inverse of format_in_list ('' un-doubles, comma-in-quotes literal)
DuckDB::format_struct \%fields ‚Üí $literal         # {a=>1,b=>2} ‚Üí {'a': '1', 'b': '2'} (DuckDB STRUCT literal, keys sorted)
DuckDB::parse_struct $literal ‚Üí \%fields          # {'a': '1', 'b': 'x'} ‚Üí {a=>"1",b=>"x"}; inverse of format_struct (quote-aware, bare null/number, '' un-doubles)
DuckDB::format_map \%pairs ‚Üí $literal             # {a=>1,b=>2} ‚Üí MAP {'a': '1', 'b': '2'} (DuckDB MAP literal, keys sorted)
```

`appender` is DuckDB's native bulk-ingest path ‚ÄĒ no SQL parse per row ‚ÄĒ and is
the fastest way to load a large dataset. Unlike `insert_many` (which takes
hashrefs and infers columns), `appender` takes an arrayref of **arrayrefs**,
each a full row in table column order:

```stryke
DuckDB::appender "events", [[1, "click"], [2, "view"], [3, "scroll"]]
```

`update` and `delete` complete the CRUD surface. `update` binds the `$set`
values (`SET col = ?, ‚Ä¶`) and interpolates `$where`; `delete` interpolates
`$where`. Both omit `$where` to affect every row and return the
affected-row count. Table and SET column names are identifier-validated;
pass trusted values in `$where`.

```stryke
DuckDB::update "events", { processed => 1 }, "id = 7"
DuckDB::delete "events", "ts < '2026-01-01'"
```

`insert_many` bulk-inserts an arrayref of hashrefs in one multi-row
INSERT. Columns are inferred from the first row's keys (sorted); every
row must share them. Table and column names are identifier-validated;
values are bound. Returns the inserted-row count.

```stryke
DuckDB::insert_many "events",
    [{ id => 1, kind => "click" },
     { id => 2, kind => "view"  }]
```

`upsert` inserts a single row and, on a unique/PK conflict over the
`conflict` columns, updates the `update` columns from the proposed row
(DuckDB `excluded.*`). The conflict-target columns must carry a UNIQUE or
PRIMARY KEY constraint. Options: `conflict => \@cols` (required); `update
=> \@cols` (defaults to every row column that isn't a conflict target ‚ÄĒ
an empty list is `DO NOTHING`); `returning => "col,‚Ä¶" | "*"` for the
affected rows instead of a count. Names are identifier-validated; values
are bound.

```stryke
DuckDB::upsert "kv", { id => 1, name => "a", hits => 1 }, conflict => ["id"]
DuckDB::upsert "kv", { id => 1, name => "x", hits => 9 },
               conflict => ["id"], update => ["hits"]   # only bump hits
my @r = DuckDB::upsert "kv", { id => 2, name => "b" },
                       conflict => ["id"], returning => "*"
```

`import` opts: `kind` (`parquet|csv|json|auto`), `replace`, plus connection.
`export` opts: `kind` (`parquet|csv|json`), `compression` (parquet only).

Statements issued with the same `%opts` run on the same cached handle,
so these ride on that affinity (no extra FFI).

```stryke
DuckDB::begin       %opts ‚Üí 1                    # BEGIN TRANSACTION
DuckDB::commit      %opts ‚Üí 1                    # COMMIT
DuckDB::rollback    %opts ‚Üí 1                    # ROLLBACK
DuckDB::transaction $code, %opts ‚Üí $code_result  # BEGIN; $code->(); COMMIT ‚ÄĒ or ROLLBACK + re-raise on die
```

### Multi-database & DDL

FFI-backed object lifecycle and multi-database operations. The file path for
`attach` rides in its own `$file` argument (mapped to `attach_path`), and
`copy_from`'s source file rides in `$file` too ‚ÄĒ neither collides with the
connection's own `db` path, so you can `attach` a second database onto an
in-memory or file-backed connection. Identifiers (alias, index, table,
columns) are validated; file paths are single-quote escaped.

```stryke
DuckDB::attach           $file, $alias, %opts ‚Üí { alias, attach_path, read_only }
                                                # ATTACH; opts: attach_read_only, if_not_exists
DuckDB::detach           $alias, %opts ‚Üí { alias, detached }            # DETACH; opts: if_exists
DuckDB::copy_from        $table, $file, %opts ‚Üí { table, path, copied } # COPY ‚Ä¶ FROM file into an existing table
                                                # opts: kind => csv|parquet|json|auto
DuckDB::create_index     $name, $table, \@columns, %opts ‚Üí { index, table, columns }
                                                # CREATE INDEX; opts: unique, if_not_exists
DuckDB::drop             $name, %opts ‚Üí { dropped, kind }   # DROP table|view|index; opts: kind, if_exists (default 1), cascade
DuckDB::drop_index       $name, %opts ‚Üí { dropped, kind }   # DROP INDEX (kind=index convenience)
DuckDB::appender_columns $table, \@columns, \@rows, %opts ‚Üí $appended_count
                                                # native Appender into a column SUBSET; unnamed cols take DEFAULT/NULL
```

`copy_from` appends a file's rows into a table that already exists (DuckDB's
native bulk file loader), distinct from `import` which does `CREATE TABLE
AS`. `appender_columns` is the column-subset companion to `appender`: name a
subset of the table's columns and supply only those values per row; every
unnamed column takes its `DEFAULT` (or NULL).

```stryke
DuckDB::attach "warehouse.duckdb", "wh", attach_read_only => 1
DuckDB::copy_from "events", "events.csv", kind => "csv"
DuckDB::create_index "idx_events_ts", "events", ["ts"], unique => 1
DuckDB::appender_columns "events", ["id", "kind"], [[1, "click"], [2, "view"]]
DuckDB::drop "events"                                  # DROP TABLE IF EXISTS events
```

## [0x05] FFI layer

Each `DuckDB::*` wrapper builds a JSON args dict and calls a sibling
`duckdb__*` symbol resolved out of `libstryke_duckdb.{dylib,so}`. The
cdylib is dlopened in-process on first `use DuckDB` (via stryke's
`pkg::commands::try_load_ffi_for` resolver hook) and caches one
`duckdb::Connection` per `(db, session, read_only)` tuple in
`OnceCell<Mutex<HashMap>>` for the life of the stryke process.

Wire shape (cdylib responses):

* `query`, `dump` ‚Üí `{"columns": [...], "rows": [{col: val, ...}, ...]}`
* `execute` ‚Üí `{"affected": <n>}`
* `exec` ‚Üí `{"ok": true}`
* `import` ‚Üí `{"table": ..., "rows": <n>}`
* `export` ‚Üí `{"path": ..., "kind": ...}`
* `tables` ‚Üí `{"tables": [...]}`
* `schema` ‚Üí `{"table": ..., "columns": [{name, type, nullable}, ...]}`
* `attach` ‚Üí `{"alias": ..., "attach_path": ..., "read_only": <bool>}`
* `detach` ‚Üí `{"alias": ..., "detached": true}`
* `copy_from` ‚Üí `{"table": ..., "path": ..., "copied": <n>}`
* `create_index` ‚Üí `{"index": ..., "table": ..., "columns": [...]}`
* `drop` ‚Üí `{"dropped": ..., "kind": ...}`
* `appender_columns` ‚Üí `{"table": ..., "columns": [...], "appended": <n>}`
* `indexes`, `constraints`, `table_info`, `database_size` ‚Üí `{"columns": [...], "rows": [...]}`
* `inspect`, `ping` ‚Üí `{...}`
* Errors ‚Üí `{"error": "<msg>"}` ‚ÄĒ the wrapper `die`s with it
